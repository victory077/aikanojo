# Issues / 要望リスト

GitHub公開後は [Issues](../../issues) で管理予定

---

## 優先度: 高

- [ ] 語尾をもっと安定させたい　ロールプレイを意識しすぎて会話が不自然な時がある（寿司のごり押し、方言から突然関西弁になるなど）
- [ ] 記憶の反映をより自然にしたい　記憶を無視した回答をすることがある
- [ ] めっちゃすぐ謝ってくるのやめてほしい　chatgptの癖が出すぎている
- [ ] キャラクター切り替え機能の完成
  - [ ] `/change`でWeb検索APIまたは外部AI連携してYAMLを自動生成
  - [ ] キャラ別にファイル名を付けて保存（例: `config/character_nezuko.yaml`）、元のキャラを上書きしない
  - [ ] `/switch`等で保存済みキャラを切り替えられるようにする
  - [x] **挨拶文のキャラ別外部ファイル化**
    - `get_time_greeting()`のセリフが莉波固定 → キャラ別に可変にする
    - character.yamlとは**別ファイル**に保存（例: `config/greetings_rinami.yaml`）
    - 理由: ①character.yamlのトークン肥大化を防ぐ ②自己生成文をシステムプロンプトに混ぜると精度低下リスク
    - キャラYAML生成後に別ステップで挨拶文を生成させる（生成プロンプトに含める）
    - 起動時はファイル読み込みのみ → ラグなし
  - ※ローカルLLMにWeb検索機能が付くか、外部API連携ができたら実装

## 優先度: 中

- [ ] 長期記憶システムにCRAG（Corrective Retrieval Augmented Generation）の導入
  - 2023年の「高知能AI彼女」プロジェクトのRAG機能を参考に、最新の技術へアップデート
  - 1. 検索された記憶ログ（短期・中期・長期）の妥当性をLLMで評価（Corrective）
  - 1. 関連性が低い記憶が引っ張られた場合は、それを破棄するか言い換え・再検索を行う
  - 1. 過去の会話の矛盾やハルシネーションを防ぎ、より自然な文脈理解を目指す

## 優先度: 低 / アイデア

- [ ] 機嫌（Mood）システムの実装
  - ユーザーの言動や放置期間などによって感情が上下するシステム
  - ロジックが複雑化しやすいため、導入の難易度が高い
- [ ] プロジェクトの方向性の最終決定: 「AI彼女」か「美少女AIエージェント」か
  - **AI彼女特化:** 感情的つながり、好感度・機嫌の変動、複雑な長期記憶と文脈理解（ロマンス重視）
  - **美少女AIエージェント:** 秘書的なタスク遂行能力、Web検索（RAG）、スケジュール管理、外部ツール（Function Calling）連携の強化（実用性重視）
